Salesforce Multi-Org Strategy

Ferramenta de apoio à decisão arquitetural para landscapes com múltiplas orgs — de 2 a dezenas.

Tem acesso à(s) org(s)? Carregue metadata para pré-preencher perguntas (opcional)

Cada org é descrita por um único arquivo JSON carregado abaixo. O motor tem 45 filtros e cada um lê campos específicos desse JSON. Você monta esse arquivo por um destes dois caminhos, mutuamente exclusivos:

Caminho A
Semi-automático (recomendado)
Você tem acesso à org via sf CLI. Um script preenche os campos técnicos (~50 campos); um segundo arquivo humano complementa o que só a área conhece (~40 campos).
Pré-requisitos: sf CLI + jq, acesso ao script extract-org-metadata.sh.
Caminho B
100% manual
Você não tem acesso à org (ou não quer usar sf CLI). Monta o JSON inteiro à mão a partir do que a área/PMO/ARB conhecem.
Pré-requisitos: nenhum. Editor de texto + o schema abaixo.

Não é obrigatório carregar nada. O questionário no modo Topologia funciona 100% sem JSON. Só o modo Alocação exige.

Caminho A — Semi-automático (sf CLI + complemento humano)

O JSON final é a fusão de duas fontes:

FonteO que preencheComo gera
MinhaOrg.json (output do script) Métricas técnicas: metadata counts, limits, PermissionSetLicense, sandboxes, installed packages, platform events, connected apps (~50 campos). ./extract-org-metadata.sh MinhaOrg MinhaOrg.json
MinhaOrg.manual.json (complemento humano) O que não sai da CLI: regulator, LGPD data controller, LOB, ApexGuru, MDM, backup/DR, team, contratos, custo, retenção (~40 campos). Copiar org-metadata.manual.template.json, renomear e editar.

O script funde os dois automaticamente ao final: MinhaOrg.manual.json tem prioridade e sobrescreve qualquer campo do script. Você só precisa carregar o .json final aqui, não o .manual.json.

Instalação dos pré-requisitos:

brew install salesforce-cli jq

Passo-a-passo:

  1. Autenticar (para sandbox: adicione -r https://test.salesforce.com). 
    sf org login web -a MinhaOrg
  2. Confirmar que está apontando pra org certa antes de extrair. 
    sf org display -o MinhaOrg
  3. Preparar o complemento humano — copie o template e edite. Deixe vazios os campos que não conhecer; eles caem em WARN, não FAIL. 
    cp org-metadata.manual.template.json MinhaOrg.manual.json
  4. Rodar o script — extrai da org e funde com MinhaOrg.manual.json se existir. Leva 1-5min. 
    chmod +x extract-org-metadata.sh
./extract-org-metadata.sh MinhaOrg MinhaOrg.json
  5. Multi-org: repita 1-4 para cada org; carregue todos os .json finais abaixo.
  6. Carregar o MinhaOrg.json em "Selecionar JSON(s) de org" no final desta seção.
Sem MinhaOrg.manual.json: o script roda mesmo assim, mas os filtros do modo Alocação que dependem de dados humanos (regulator, LGPD, ApexGuru, backup, team, MDM, contratos) ficam em WARN. Um aviso explícito é impresso ao final. É legítimo rodar assim para preview técnico, mas para decisão final o complemento humano é o que dá poder ao motor.
Se o script falhar: bash -x ./extract-org-metadata.sh MinhaOrg MinhaOrg.json 2>&1 | tail -40. Causas comuns: usuário sem Tooling API, org sem Package2, sandbox onde SandboxInfo não é consultável.
Caminho B — 100% manual (sem sf CLI)

Sem acesso à CLI, você monta um arquivo .json por org com os dados que a área/PMO/ARB conhecem. Não use o script nem o template do complemento; use o schema unificado abaixo como base.

Passo-a-passo:

  1. Copie o exemplo mínimo abaixo e salve como MinhaOrg.json.
  2. Ajuste os valores. Consulte a tabela Schema completo por bloco abaixo para adicionar mais campos.
  3. Carregue o arquivo em "Selecionar JSON(s) de org".

Exemplo mínimo funcional — funciona para o modo Alocação, mas gera muitos WARN por causa dos campos não declarados. Todos os campos abaixo são obrigatórios apenas se você quer PASS nos filtros que os consomem.

{
  "orgName": "MinhaOrg",
  "edition": "Enterprise",
  "regulator": "NONE",
  "dataControllerLGPD": "Minha Empresa S.A.",
  "lobOwner": "B2C / Vendas Diretas",
  "orgType": "production",
  "hasPersonAccount": false,
  "isMultiCurrency": false,
  "storagePct": 40,
  "apiUsagePct": 25,
  "customObjectLimitPct": 15,
  "releaseSchedule": "biweekly",
  "shieldEnabled": false,
  "fatEnabled": false,
  "supportTier": "premier"
}
Regra de ouro: cada campo omitido vira WARN no filtro que o consome, nunca PASS silencioso. Se estiver em dúvida se sabe o valor, deixe o campo fora — a diagnose fica correta.
Schema completo — todos os campos que o motor consome, agrupados por bloco

Referência única, usada nos dois caminhos. Cada bloco corresponde a uma seção do formulário do processo e alimenta um grupo de filtros. Em Caminho A, "coleta técnica" = script; "complemento humano" = .manual.json. Em Caminho B, tudo é escrito à mão no mesmo arquivo.

Valores válidos para enums e campos textuais:

  • regulator: NONE, LGPD, BACEN, CVM, SUSEP, PREVIC, ANS, ANEEL, ANATEL, GDPR, CCPA, HIPAA, OTHER
  • orgType: production, sandbox
  • dataModel: b2b, personAccount, fsc, healthcloud, nonprofit, education, manufacturing, custom
  • releaseSchedule: weekly, biweekly, monthly, quarterly
  • envStrategy: canary, staged, prod-only, bigbang
  • apexGuruTrend: improving, stable, stable-on-debt, degrading
  • supportTier: standard, premier, signature
  • sharingModel: private, public, controlled
  • backupFrequency: hourly, daily, weekly
  • installedPackages[].licenseType: string livre; regra especial: se contiver oem (case-insensitive), o filtro packages vira WARN quando o processo modifica esse namespace.
  • installedPackages[].allowsExtension: true ou false. Se false, processo que tenta modificar o package vira FAIL.
  • interOrgLatencyMsP95: objeto {"NomeDaOrg": 250} em ms.
  • availableLicenses: objeto {"Salesforce": 120, "Platform": 300}.
  • platformEventPublishedTypes[]: {"name":"AccountCDC","type":"cdc|standard|high-volume"}.

Exemplos completos em samples/: OrgVarejoPF.json (B2C regulado — Person Account, HVPE, Pub-Sub), OrgAsset.json (mercado de capitais — B2B, CVM, sem HVPE), OrgCorporativa.json (org de plataforma B2B — capacidade apertada, muitos add-ons). Use como referência ao montar orgs no Caminho B — copie o perfil mais próximo do seu setor e adapte.

O arquivo é processado apenas no seu navegador. Nada é enviado a nenhum servidor.
